# Design: Insider-Trade Data Source Redesign (v0.2)

**Date:** 2026-04-24
**Supersedes source layer of:** [`2026-04-23-market-digest-skeleton-design.md`](2026-04-23-market-digest-skeleton-design.md)
**Status:** Draft — awaiting user review before implementation plan.
**Prior brief:** [`docs/handoff-2026-04-24-data-source-redesign.md`](../../handoff-2026-04-24-data-source-redesign.md)

## Context

v0.1.1 shipped with `timothycarambat/senate-stock-watcher-data` as the insider-trade source. That repository was abandoned upstream on 2020-12-05, so the pipeline was ingesting 5-year-old trades and rendering them with today's timestamp in the TUI — stale data masked as fresh. All three `insiders:` sources in `config/sources.yml` are currently `enabled: false`; the local SQLite file has been wiped; the reusable Go packages (`internal/insiders/{hash,store,rules}.go`, `jobrun`, `alert`, `data`) remain untouched and tested.

v0.2 replaces the data source(s) with fresh, legitimate feeds, adds schema capacity for SEC Form 4's richer data shape, and fixes the TUI behaviors that made the staleness bug silent (default landing screen, absent scroll feedback, missing transaction date in the list view).

## Scope

### In scope (v0.2)

1. **SEC Form 4 fetcher** — corporate insider trades (directors, officers, 10%-owners). Watchlist-scoped, free, real-time, official.
2. **Finnhub Congressional-trading fetcher** — politician trades via Finnhub's free-tier endpoint. Watchlist-scoped.
3. **QuiverQuant Congressional-trading fetcher** — same domain as Finnhub, paid API. Written but **disabled by default**; activating = flip `enabled: true` in `sources.yml` and set `QUIVER_API_KEY`. Zero code changes needed to activate later.
4. **Schema extension** — four nullable columns on `insider_trades` to hold Form 4's extra signal (shares, price-per-share, transaction code, security type).
5. **Alert-rule filter on Form 4 transaction codes** — only `P` (open-market purchase) and `S` (open-market sale) fire alerts; all other codes still *stored* for drill-down but don't generate noise (RSU vestings, option grants, tax withholdings).
6. **Minimum-viable TUI fixes** — default-to-Watchlist (not Alerts), shared viewport wrapper with PgUp/PgDn/Home/End, date-display fix on the Alerts screen (both `transaction_ts` and `created_ts` shown).
7. **Config cleanup** — dead `senate:` and `house:` keys removed from `sources.yml`.

### Out of scope — deferred with rationale

| Item | Deferred to | Reason |
|---|---|---|
| Senate EFD direct scraper | v0.3+ (maybe never) | Session-cookie consent gate + PDF parsing at the leaf. Weeks of engineering + ongoing maintenance for marginal value over Finnhub/Quiver. |
| House Clerk `FD.zip` scraper | v0.3+ (maybe never) | Same PDF-parsing problem at the leaf. Same cost/benefit calculus. |
| Watchlist screen enrichment (net $, largest trade, latest txn date, distinct filer count) | v0.2.5 | TUI is "minimum viable" in v0.2 per user direction. Schema supports the enrichment already. |
| Multi-column ticker detail view (role, security type, code, shares, price columns) | v0.2.5 | Same. |
| Cross-source dedup (Finnhub ↔ Quiver overlap) | v0.3 | Only matters if both are active simultaneously. v0.2 has Finnhub active and Quiver dormant, so no overlap in practice. See "Risks" section for suggested approach. |
| Historical backfill beyond 7 days | Not planned | User explicit: "I only care about what's new." Escape hatch: bump `max_lookback_days` in `sources.yml`. |

### Watermark resolution

No watermark table. Watermark = `MAX(filing_ts)` from `insider_trades` for a given `(source, ticker)` pair, computed at query-time. The fetcher's effective cutoff is:

```
cutoff = max(now - max_lookback_days * 86400, COALESCE(watermark, 0))
```

On an empty DB with `max_lookback_days: 7`, every fetch looks back exactly 7 days. Once trades exist, incremental fetches only re-pull filings newer than the latest stored filing for that ticker, bounded by the 7-day ceiling. Skipping a month means losing 3 weeks of middle activity — explicit acceptance per design discussion.

## Schema

### Migration `migrations/0003_form4_columns.sql`

```sql
ALTER TABLE insider_trades ADD COLUMN shares INTEGER;
ALTER TABLE insider_trades ADD COLUMN price_per_share REAL;
ALTER TABLE insider_trades ADD COLUMN transaction_code TEXT;
ALTER TABLE insider_trades ADD COLUMN security_type TEXT;

CREATE INDEX IF NOT EXISTS insider_trades_txcode
  ON insider_trades(transaction_code) WHERE transaction_code IS NOT NULL;
```

**Column semantics:**

| Column | Type | Form 4 | Finnhub/Quiver |
|---|---|---|---|
| `shares` | INTEGER | share count (signed: +acquired, −disposed) | NULL |
| `price_per_share` | REAL | execution price USD | NULL |
| `transaction_code` | TEXT | `P` / `S` / `A` / `M` / `F` / `G` / `D` / ... (single-char SEC code) | NULL |
| `security_type` | TEXT | `common` or `derivative` | NULL |

For Form 4, `amount_low = amount_high = round(abs(shares) * price_per_share)` — giving rules + TUI a single dollar axis across all sources. The existing `role` TEXT column holds the filer's role for both Form 4 (`CEO`, `CFO`, `Director`, `10%-Owner`) and politicians (`Senator`, `Representative`) — no separate `relationship` column.

The partial index on `transaction_code` enables fast "only open-market purchases" rule queries without penalizing the hundreds of thousands of politician rows that will have NULL in that column.

## Hashing / dedup strategy

The existing `Hash(Trade) string` keys dedup on `source | filer | ticker | transaction_ts | amount_low | amount_high | side`. That's right for politician data (unstructured filings with range-based dollar amounts). It's wrong for Form 4 because:

1. **Same-day multi-transaction filings collapse.** A CEO doing an option exercise (M) *and* an open-market sale (S) on the same day for the same $ would dedup to one row under the existing hash — losing half the filing.
2. **Form 4 has a true unique identity.** Every EDGAR filing has an accession number (e.g., `0001127602-26-000123`); each filing contains 1..N indexed transactions. `(accession, index)` is SEC's own guaranteed-unique key.

Per the handoff's "Hash is load-bearing" rule, we do *not* modify `Hash()`. Instead:

```go
// internal/insiders/hash.go — NEW sibling function
func HashForm4(accession string, txnIndex int) string {
    h := sha1.Sum([]byte(fmt.Sprintf("sec-form4|%s|%d", accession, txnIndex)))
    return hex.EncodeToString(h[:])
}
```

The Form 4 fetcher calls `t.Hash = HashForm4(accession, i)` during ingestion; Finnhub/Quiver fetchers keep calling `t.Hash = Hash(t)`. Both SHA1 hex outputs fit the existing `hash TEXT UNIQUE` column; `StoreInserts` remains source-agnostic.

Accession number is extractable from the Form 4 filing URL (path format: `.../Archives/edgar/data/{cik}/{accession-no-dashes}/{accession-with-dashes}-index.htm`). The Form 4 fetcher parses it during ingestion and stores the canonical dashed form as part of the existing `raw_url` column — no new schema column needed.

### CIK resolution

SEC publishes a ticker→CIK mapping at `https://www.sec.gov/files/company_tickers.json` (~12k rows, ~900KB). Strategy:

- First Form 4 fetch: download, write to `data/cik_cache.json`.
- Subsequent fetches: load from disk; if file mtime is > 30 days, refresh.
- Ticker not in SEC list (ADRs, OTC, foreign issuers): log warning, skip that ticker for Form 4 only (Finnhub/Quiver still try).
- Cache file format: SEC's native JSON shape, no reshaping. Fetcher builds an in-memory `map[string]string` on load.

## Alert rules — Form 4 transaction-code filter

The single largest signal-quality lever. Without this filter, Form 4 spams alerts for RSU vestings, option grants, and tax withholdings — none of which represent insider conviction.

### Changes in [`internal/insiders/rules.go`](../../../internal/insiders/rules.go)

**1. Per-trade loop** — skip alert generation (but not storage) for non-open-market Form 4 codes:

```go
for _, t := range newTrades {
    // NEW: suppress alerts for mechanical Form 4 transaction codes.
    // The row is still in insider_trades; it just doesn't alert.
    if t.Source == "sec-form4" && t.TransactionCode != "P" && t.TransactionCode != "S" {
        continue
    }
    // ... existing watchlist_hit / amount threshold logic unchanged ...
}
```

**2. Cluster rule** — extend the SQL WHERE clause so the "3+ distinct filers in 7d" rule only counts open-market Form 4 activity:

```sql
SELECT ticker, COUNT(DISTINCT filer) AS filers
FROM insider_trades
WHERE ticker IS NOT NULL AND ticker != '' AND transaction_ts >= ?
  AND (source != 'sec-form4' OR transaction_code IN ('P', 'S'))
GROUP BY ticker
HAVING filers >= 3
```

Politicians (where `source` is `finnhub`/`quiver`) always pass the first disjunct; Form 4 rows must have code `P` or `S`.

### Transaction-code reference

For future maintainers and for the spec's self-documentation, the SEC codes we explicitly *suppress* from alerts:

| Code | Meaning | Why suppressed |
|---|---|---|
| `A` | Grant, award, or other acquisition | Mechanical — insider didn't choose to buy |
| `M` | Exercise/conversion of derivative | Often followed by same-day S; not a conviction signal on its own |
| `F` | Tax withholding on vested equity | Purely mechanical |
| `G` | Bona fide gift | Not a trading decision |
| `D` | Sale/disposition to issuer (buyback) | Issuer action, not insider conviction |
| `I`, `J`, `K`, `L`, `U`, `V`, `W`, `X`, `Z` | Various mechanical / unusual | Rare; conservative to suppress |

Codes we alert on: `P` (open-market purchase) and `S` (open-market sale) only.

If the user later wants finer-grained control (e.g., surface `M+S` same-day as a distinct "option cashout" signal), it's a post-v0.2 rules-engine enhancement.

## TUI changes

### Default landing screen

[`dashboard/main.go:151`](../../../dashboard/main.go#L151): `current: screenAlerts` → `current: screenWatchlist`. The Alerts screen remains accessible via the `1` key (its existing shortcut).

Rationale: the user's actual question is "who's buying what across my watchlist," not "triage each alert." The email-inbox paradigm was the wrong default; the Watchlist screen (one row per ticker with activity counters, drill-down via Enter) is the right landing page.

### Shared viewport wrapper — new `dashboard/internal/viewport/` package

Responsibility: given a slice of rendered rows (`[]string`), a cursor index, and a window height, produce the correctly-scrolled visible slice plus the cursor-position indicator. Handle keyboard navigation.

**Public surface:**

```go
package viewport

type Model struct {
    rows   []string
    cursor int
    top    int
    height int
}

func New() Model
func (m Model) SetRows(rows []string) Model
func (m Model) SetHeight(h int) Model
func (m Model) Cursor() int
func (m Model) Update(msg tea.Msg) (Model, tea.Cmd)  // handles nav keys
func (m Model) View() string                         // returns visible slice, cursor rendered
```

**Key bindings (handled inside `Update`):**

| Key | Action |
|---|---|
| `↑` / `k` | cursor − 1 |
| `↓` / `j` | cursor + 1 |
| `PgUp` / `ctrl+u` | cursor − (height / 2) |
| `PgDn` / `ctrl+d` | cursor + (height / 2) |
| `Home` / `g` | cursor = 0 |
| `End` / `G` | cursor = len(rows) − 1 |

**Scroll math (invariant: cursor always visible):**

```
if cursor < top:        top = cursor
if cursor >= top+usableHeight:  top = cursor - usableHeight + 1
```

Where `usableHeight = height - header_lines - footer_lines`.

### Screen embedding

Each of `AlertsModel`, `WatchlistModel`, `TickerModel`, `JobsModel` embeds `viewport.Model`, renders all rows to a `[]string`, hands it to the viewport, delegates nav-key handling to the viewport, and composes the viewport's `View()` with the screen's own header + footer. Each screen retains its data-loading logic and its screen-specific actions (e.g., Alerts still owns `x` for mark-seen, Watchlist still owns `Enter` for drill-down).

### Alerts date-display fix — [`dashboard/internal/screens/alerts.go`](../../../dashboard/internal/screens/alerts.go)

Row format before:

```
  2026-04-24 07:31  insiders  watch    NVDA    Pelosi buy NVDA ($50001-$100000)
```

Row format after:

```
  txn 2026-04-10  alert 04-24 07:31  insiders  watch    NVDA    Pelosi buy NVDA ($50001-$100000)
  txn —           alert 04-24 07:35  insiders  watch    NVDA    Cluster: 4 filers on NVDA in last 7d
```

- Transaction date leads (the staleness signal).
- Alert-creation time is abbreviated (`MM-DD HH:MM`) to conserve horizontal budget.
- Cluster alerts and other rule-fired events without a single `transaction_ts` show `txn —`.

### Data plumbing for the date fix

`internal/data.AlertRow` gains a nullable field:

```go
type AlertRow struct {
    // existing fields...
    TransactionTS *int64  // from alerts.payload JSON, nil for cluster/non-trade alerts
}
```

`RecentAlerts` query extracts it:

```sql
SELECT id, source, severity, ticker, title, body, created_ts, seen_ts,
       CAST(json_extract(payload, '$.transaction_ts') AS INTEGER) AS transaction_ts
FROM alerts
ORDER BY created_ts DESC
LIMIT ?
```

SQLite's `json_extract` returns NULL for missing keys, which scans into `*int64(nil)` via `sql.NullInt64` unwrapping. Backward-compatible — alerts inserted before the payload convention will simply show `txn —`.

The per-trade `EvaluateRules` already stores `transaction_ts` in `alerts.payload` ([`internal/insiders/rules.go:66`](../../../internal/insiders/rules.go#L66)), so no change needed to alert insertion.

### Expected ICHR behavior specifically

- `ICHR`'s Form 4 volume is typically 0–3 filings per week → 7-day fetch often returns zero rows.
- Finnhub Congressional activity on ICHR is typically zero (politicians rarely trade mid-caps).
- This makes ICHR a clean "pipeline runs with no data" verifier, complementing NVDA's "pipeline runs with real data" case.

## Operational notes

- **`SEC_USER_AGENT` is required.** The Form 4 fetcher validates at startup: env var must be set *and* must not match the placeholder string `market-digest <your-email@example.com>`. Fail loud with a clear message pointing at README / CLAUDE.md. SEC throttles/blocks requests without a contact UA.
- **`FINNHUB_API_KEY` is optional.** If unset while `finnhub.enabled=true`, the fetcher logs `[finnhub] FINNHUB_API_KEY unset; skipping source` and proceeds with the other enabled sources.
- **`QUIVER_API_KEY` is unused unless `quiver.enabled=true`.** When that flips, the same "required-or-skip" logic as Finnhub applies (but log message directs at Quiver signup).
- **SEC rate limits:** 10 req/s aggregate across all SEC endpoints per user agent. The Form 4 fetcher polites itself with ≤5 req/s (200ms sleep between calls) as a conservative default. For a 20-ticker watchlist + ~5 filings per issuer, total wall-clock is under 30s.
- **PR shape:** single PR for v0.2 — schema + rules + TUI are tightly coupled; the spec is reviewed as one unit.

## Risks and open concerns

### Risk 1 — Finnhub coverage / freshness

Finnhub's free-tier Congressional data has historically lagged Quiver by days-to-weeks and had missing filers. If the user's real ideation use-case demands freshness, Finnhub may feel thin. **Mitigation:** Quiver infra is ready — flipping `enabled: true` + setting `QUIVER_API_KEY` is a one-line config change away.

### Risk 2 — Cross-source dedup (deferred)

If Finnhub AND Quiver are both enabled, the same Pelosi trade may produce two rows with slightly different amount rounding (e.g., Finnhub reports `amount_high=100000`, Quiver reports `amount_high=99999`). Both produce different `Hash()` outputs → both inserted → cluster rule's `COUNT(DISTINCT filer)` double-counts.

**Suggested v0.3 approach (not implemented):** a post-fetch dedup pass that runs BEFORE `StoreInserts`:

```go
// Collapse Finnhub/Quiver pairs that share (filer, ticker, transaction_date, side)
// with amount_high within 5% of each other. Prefer the Quiver row (richer data).
```

For v0.2 with Quiver dormant, this is a noop concern. A comment in the spec + an issue in the repo is sufficient.

### Risk 3 — Form 4 `ownership.xml` schema drift

SEC has changed the Form 4 XML schema before (rarely, but historically). Golden-file tests pin the parser to a specific schema version; a real-world drift would break parsing.

**Mitigation:** the fetcher logs and skips (not fails) filings that fail to parse, so a schema drift on one filing doesn't take down the whole run. Parser errors surface in the `job_runs` row's error notes.

### Risk 4 — Watermark resumption after long idle

If the user doesn't run the fetcher for a month, they explicitly accept losing 3 weeks of middle data (per design discussion). The 7-day ceiling is the protection against an accidental "fetch everything since 2020" catch-up.

### Risk 5 — TUI viewport refactor introduces regressions

Adding a shared viewport across four screens touches all of them. Risk: subtle behavior change (e.g., existing `x` key on Alerts clashes with viewport key handling).

**Mitigation:** each screen's `Update` delegates to viewport *first*, then handles its own screen-specific keys only if viewport returned an unhandled message. Unit tests on the viewport itself; per-screen tests updated to cover both nav and screen-specific keys.
